## 오늘 회의에서 얘기할 것들

- 주제
    
    ### 현재 문제
    
    - 역할 분담 방식이 아직 합의되지 않았다
    - 섹션별로 나누자는 의견이 있는데, 우리 프로젝트 구조상 충돌이 생길 수 있다
    - 기획 방향을 각자 다르게 생각하고 있을 수 있다
    - 의견이 충돌했을 때 누가 최종 결정하는지 정해져 있지 않다
    
    ---
    
    ### 오늘 결정해야 할 것들
    
    **1. 기획 방향 먼저 맞추기** 우리가 이 프로젝트를 왜 만드는지, 각자 생각을 꺼내놓고 하나로 맞추기
    
    **2. 역할 분담 방식
    
    -> 섹션기준으로할꺼면 섹션은 어떻게 나눌것인가 / 데이터는 어떤 데이터를 사용할것인가 정의 필요
    
    - 역할분담 방식
        
        ### 1. 기능 기준 분업
        
        "지도는 내가, 타임라인은 네가" 처럼 기능 단위로 나누는 방식이에요.
        
        - **장점** — 자기가 맡은 기능에 깊게 집중할 수 있고, 완성도가 높아지기 쉬워요
        - **단점** — 기능끼리 연결되는 지점에서 충돌이 생기기 쉽고, 한 기능이 늦어지면 전체가 늦어져요
        
        ### 2. 레이어 기준 분업
        
        "JS는 내가, CSS는 네가, 데이터는 네가" 처럼 기술 층위로 나누는 방식이에요.
        
        - **장점** — 파일 충돌이 거의 없고, 책임소재가 명확해요
        - **단점** — 레이어 간 소통이 잘 안 되면 연결이 어긋날 수 있어요
        
        ### 3. 역할 기준 분업
        
        개발자·디자이너·기획자처럼 직군 개념으로 나누는 방식이에요.
        
        - **장점** — 각자 잘하는 걸 하니까 효율이 높아요
        - **단점** — 팀원 역량이 비슷하면 경계가 애매해져요
        
        ### 4. 섹션 기준 분업
        
        "1~3섹션은 내가, 4~6섹션은 네가" 처럼 페이지 구간으로 나누는 방식이에요.
        
        - **장점** — 직관적이고 나누기 쉬워요
        - **단점** — 공통 요소가 많은 프로젝트에서는 충돌이 잦아요. 우리 프로젝트엔 맞지 않아요
        
        ### 5. 단계 기준 분업
        
        "1차 개발은 A가, 2차 고도화는 B가" 처럼 시간 순서로 나누는 방식이에요.
        
        - **장점** — 순서가 명확해서 충돌이 없어요
        - **단점** — 앞 사람이 늦으면 뒷 사람이 놀게 돼요. 장기 프로젝트엔 비효율적이에요
        
        ### 6. 혼합 방식
        
        위 방식들을 섞어서 쓰는 거예요. 예를 들어 **레이어 기준 + 역할 기준**을 합치면 "JS 담당이지만 디자인 결정은 CSS 담당이 한다" 처럼 운영할 수 있어요.
        
        - **장점** — 프로젝트 성격에 맞게 유연하게 조합할 수 있어요
        - **단점** — 합의가 복잡해질 수 있어서 초반에 잘 정해놔야 해요
    
    **3. 영역별 책임권자 정하기** 의견 충돌 시 누가 최종 결정하는지 명확히 하기
    
    **4. 문서 작업 분담 정하기** 기획의도, 유사사례, 개발서 등 누가 어떤 문서를 맡을지
